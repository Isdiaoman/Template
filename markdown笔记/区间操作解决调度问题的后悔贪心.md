# 区间操作解决调度问题的后悔贪心

> 用线段树维护区间信息，实现可撤销 / 可反悔的贪心调度。

## 问题模型

有 $m$ 个任务，每个任务有一个难度值 $a_i$。每个工人有一个能力值，能力值 $\geq$ 任务难度时才能完成该任务，且每个工人最多分配一个任务。

现在有 $q$ 次操作，每次操作**新增**一个能力值为 $b_i$ 的工人。每次操作后，需要立即输出：当前所有工人最多能完成多少个任务。

## 核心思想

如果没有动态添加工人，这就是一个简单的贪心匹配问题。但问题要求**在线**维护答案——工人不断加入，我们需要随时调整之前的分配策略。"手动"重新调度所有工人是困难的，但换一个维度看待这个问题，就变得简单了。

**定义**：记 $L_i$ 为「能力值 $\geq i$ 的**空闲**工人数量」。

现在考虑一个任务（难度 $y$）被完成时会怎样：需要从空闲工人中选一个能力值 $\geq y$ 的人来干。这意味着**所有** $i \le y$ 的 $L_i$ 都会减少 $1$——因为能力值 $\geq y$ 的空闲工人少了一个，他对所有 $i \le y$ 的 $L_i$ 都有贡献。

用公式表达：
- **完成任务**（难度 $y$）：$L[0..y]$ 全部 $-1$
- **新增工人**（能力 $x$）：$L[0..x]$ 全部 $+1$

这恰好是**区间加**操作，可以用线段树维护。

那么怎么判断当前状态是否可行？显然 $L_i$ 表示的是"空闲工人数量"，所以任何时候必须 $L_i \ge 0$ 对所有 $i$ 成立。如果 $\min L_i < 0$，说明出现了"负数空闲工人"，这是不可能的，意味着当前分配方案不可行。

因此只需线段树维护区间最小值，就能快速判断状态是否可行。

## 算法流程

1. 初始所有 $L_i = 0$，已完成任务数 $ans = 0$。
2. 将任务按难度从小到大排序。
3. 按难度从小到大的顺序，尝试贪心匹配下一个任务：
   - 在线段树上将 $L[0..y]$ 全部 $-1$
   - 查询全局 $\min L_i$
   - 若 $\min L_i \ge 0$：分配成功，$ans \gets ans + 1$
   - 若 $\min L_i < 0$：**回滚**（$L[0..y]$ 全部 $+1$），$ans$ 不变
4. 每次有工人加入时：
   - $L[0..x]$ 全部 $+1$
   - 重新尝试第 3 步中之前回滚过的任务（因为现在可能可以分配了）
5. 每次操作后输出 $ans$。

本质上是让线段树维护"空闲工人数的前缀信息"，利用区间 $\min$ 快速判断是否还能分配，来实现可反悔的贪心。

## 适用范围

凡是需要**在线动态调整分配方案**的调度问题，如果能将约束转化为"某种前缀量必须 $\ge 0$"的形式，就可以尝试这种线段树 + 后悔贪心的做法。
